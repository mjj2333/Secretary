# Secretary Phase 5.5 — Web Push + Offline Behavior Design

**Date:** 2026-06-01
**Status:** Approved design, ready for implementation plan
**Builds on:** Phase 2.5 PWA foundation (SW/manifest via vite-plugin-pwa, SSE plumbing), the designed screens (draft loop), Phase 5 drafting (`draft:ready` event), Phase 2 push stubs.

## Overview

Deliver BRIEF §5.5: a **Web Push** notification when a draft becomes ready (tap → opens that thread), with **quiet hours**, plus minimal **offline behavior** — an offline send-queue and an offline banner. The headline acceptance criterion (BRIEF §5.5): _a new draft → the device gets a Web Push within ~30s → tapping it opens the PWA to that thread._

## Scope

**In scope:**

- **A. Web Push end-to-end** — VAPID keypair generated on first run (→ keychain); expose the public key; a `WebPushSender` that fires on `draft:ready`; SW `push` + `notificationclick` handlers (notification → open `/threads/:id`); replace the `POST /push/test` 409 stub with a real send.
- **B. Quiet hours** — a settings window (default 22:00–08:00 server-local) that suppresses _push_ only; drafts still generate and SSE still fires.
- **C. Offline send-queue (minimal)** — SW BackgroundSync queues `POST /api/v1/drafts/:id/send` when offline and replays on reconnect; the UI shows "Queued" (keyed off `navigator.onLine`) rather than "Sent".
- **E. Offline banner** — "Offline — last synced <ago>" in the app shell, driven by `navigator.onLine` + a `lastSyncAt` updated on SSE/successful queries.
- SW strategy switch to **`injectManifest`** (custom `src/sw.ts`) to host push + background-sync + precache in one place.

**Deferred (not this phase):**

- **D. Runtime API caching** (stale-while-revalidate of `GET /api/v1`) — risks stale threads/drafts; SSE already provides freshness. Explicitly out.
- Full offline _compose_ (queuing the body PATCH as well as the send) — only the send request is queued.
- The phone-facing trusted-HTTPS deployment (Cloudflare tunnel / LAN cert) — a separate deployment concern; the push mechanism is verified on the desktop browser.
- A polished notifications/permissions onboarding UI — a single "Enable notifications" control in Settings is enough.

## Architecture

### Server (`apps/service`)

Add `web-push` as a dependency.

**1. VAPID lifecycle (first-run, idempotent).** At startup, read `app.vapid-public-key` / `app.vapid-private-key` from the keychain (`KeychainStore`, service `secretary`). If absent, generate via `webpush.generateVAPIDKeys()` and store both; the VAPID subject is the fixed constant `mailto:secretary@localhost`. When keys exist, call `webpush.setVapidDetails(subject, public, private)`. Expose via:

- `GET /api/v1/push/vapid-public-key` → `{ data: { publicKey } }` (behind the normal auth guard). Returns the configured public key, or a `push_not_configured` (409) if somehow absent.

**2. `WebPushSender`** (`server/agent/WebPushSender.ts`). Constructed with `{ subscriptions: PushSubscriptionRepository, threads: ThreadsRepository, messages: MessagesRepository, contacts: ContactsRepository, settings: SettingsRepository, sender: WebPushClient, now: () => Date }`. Subscribes to `eventBus` `draft:ready` (payload `{ threadId, draftId, accountId }`). On event:

- If quiet hours active (per settings + `now`) → return (no send; drafts/SSE unaffected).
- Resolve `senderName` for `threadId` via the shared `resolveSenderName` (thread row + messages + contacts).
- Build payload `{ title: "New draft ready for <senderName>", body: <thread subject ?? "">, data: { url: "/threads/<threadId>" } }`.
- For each stored subscription, `sender.send(subscription, JSON.stringify(payload))`. On a `404`/`410` (gone), `subscriptions.deleteByEndpoint(endpoint)` (prune). Other errors are logged (metadata only — never the payload/body) and skipped.
- Never throws (mirrors the Drafter/Classifier never-throws contract; wired as an event listener).
  `sender: WebPushClient` is a thin interface (`send(sub, payload): Promise<void>` that throws an error carrying `statusCode`) wrapping `webpush.sendNotification`, so the sender is unit-testable with a fake.

**3. Quiet hours** — pure helper `isQuietHours(now: Date, start: string, end: string): boolean` (`server/agent/quietHours.ts`), `HH:MM` 24h, handling the overnight wrap (start > end). Settings keys `notifications.quiet_hours_start` (default `"22:00"`), `notifications.quiet_hours_end` (default `"08:00"`), seeded in `seed.ts`. Uses server-local time (the desktop is in the principal's timezone).

**4. `POST /push/test`** — replace the stub: 409 `push_not_configured` if VAPID absent; otherwise send a fixed test notification (`{ title: "Secretary test", body: "Push is working.", data: { url: "/needs-attention" } }`) to all subscriptions, pruning gone ones; return `{ data: { sent: <count> } }`.

**5. Wiring (`index.ts`)** — after the eventBus + repos exist and VAPID is configured, construct `WebPushSender` and let it subscribe to the eventBus. If VAPID keys are absent, skip (log a warn), consistent with the gateway-absent pattern.

### Client (`apps/service/pwa`)

- **`src/push/subscribe.ts`** — `enablePush(): Promise<'subscribed' | 'denied' | 'unsupported'>`: bail `unsupported` if no `serviceWorker`/`PushManager`/`Notification`; `Notification.requestPermission()` → if not granted, `denied`; get the SW registration; `GET /push/vapid-public-key`; `registration.pushManager.subscribe({ userVisibleOnly: true, applicationServerKey: urlBase64ToUint8Array(publicKey) })`; `POST /push/subscribe` with `{ endpoint, keys: { p256dh, auth }, userAgent }` (from `subscription.toJSON()`); → `subscribed`. Pure helper **`urlBase64ToUint8Array`** (TDD'd).
- **Settings UI** (`src/routes/Settings.tsx`) — add a "Notifications" section above the existing settings dump: an **Enable notifications** button calling `enablePush`, reflecting state (`Notification.permission` / result), and a **Send test** button (`POST /push/test`). Keep the rest of the placeholder.
- **Offline indicator** — `src/util/useOnlineStatus.ts` (`navigator.onLine` + `online`/`offline` listeners) and a `lastSyncAt` updated whenever an SSE message arrives or a query succeeds (a tiny module-level store or context). `AppShell` renders a banner **"Offline — last synced <formatTimeAgo(lastSyncAt)>"** when offline.
- **Send "Queued" UX** — in `ThreadView`'s send flow: if the send mutation fails **and** `!navigator.onLine`, show "Queued — will send when you're back online" instead of an error banner (the SW's BackgroundSync does the real replay). Online failures keep the existing error banner.

### Service worker (`apps/service/pwa/src/sw.ts`, injectManifest)

- `precacheAndRoute(self.__WB_MANIFEST)`; navigation fallback to `index.html`; `self.skipWaiting()` + `clientsClaim()` (preserve today's auto-update behavior).
- `registerRoute(({url, request}) => request.method === 'POST' && /\/api\/v1\/drafts\/[^/]+\/send$/.test(url.pathname), new NetworkOnly({ plugins: [new BackgroundSyncPlugin('send-queue', { maxRetentionMinutes: 24*60 })] }), 'POST')` — the offline send-queue.
- `self.addEventListener('push', ...)` → parse JSON → `showNotification(title, { body, data, tag: 'secretary', icon: '/icon.svg' })`.
- `self.addEventListener('notificationclick', ...)` → `notification.close()`; focus an existing client on `data.url` if open, else `clients.openWindow(data.url)`.
- **`vite.config.ts`** — switch VitePWA to `strategies: 'injectManifest'`, `srcDir: 'src'`, `filename: 'sw.ts'`, `injectManifest: { globPatterns: ['**/*.{js,css,html,svg}'] }`; keep the existing `manifest`. (Output is still `dist/sw.js`, served by the service as today.) Add the `injectRegister`/types as needed so `main.tsx`'s `registerSW` still works.

### Data flow

Drafter writes a draft → emits `draft:ready` → **(a)** SSE invalidates an open PWA's queries (existing) and **(b)** `WebPushSender` sends a push (unless quiet hours) → push service → SW `push` → notification → tap → `notificationclick` opens/focuses `/threads/:id`. Offline send: tap Send while offline → SW BackgroundSync queues `POST /send` → page fetch rejects → UI shows "Queued" → on reconnect the SW replays the request (with its original bearer header) → delivered.

### Error / edge cases

- **VAPID absent**: `GET /push/vapid-public-key` and `POST /push/test` → 409 `push_not_configured`; `WebPushSender` not constructed (warn at boot). The PWA's `enablePush` surfaces the 409.
- **Permission denied / unsupported**: `enablePush` returns `denied`/`unsupported`; nothing stored; Settings shows the state.
- **Gone subscription** (404/410 from the push service): pruned from `push_subscriptions`.
- **Quiet hours**: push suppressed; draft + SSE unaffected.
- **Offline**: only `navigator.onLine`-driven banner + the queued send; no API GET caching (deferred D), so lists simply fail to their existing empty/error states offline.
- **Security (BRIEF §5)**: never log notification bodies/subjects/payloads — only metadata (counts, endpoints hashed or omitted, statusCodes). The push payload carries the subject + sender name (necessary for a useful notification) but is sent only to the principal's own registered subscriptions; it is not logged.

### Testing (per BRIEF §18 — logic TDD; UI/SW manual)

- **Server (TDD)**: `isQuietHours` (pure, incl. overnight wrap + boundary); `WebPushSender` with a fake `WebPushClient`/repos/clock — sends to all subscriptions, prunes on 404/410, skips during quiet hours, resolves senderName into the title; VAPID generate-once idempotence (fake keychain). Routes via `app.inject`: `GET /push/vapid-public-key` (configured → key; absent → 409); `POST /push/test` (sends to subscriptions / 409 when unconfigured).
- **PWA (TDD)**: `urlBase64ToUint8Array`; `useOnlineStatus` / `lastSync` where cheap.
- **Manual (desktop browser, runbook)**: Settings → Enable notifications (grant) → subscription stored; generate a draft → notification appears (title "New draft ready for …") → click opens the thread; `POST /push/test` works; set quiet hours covering "now" → no push but draft/SSE still happen; go offline → tap Send → "Queued" → reconnect → it sends; offline banner shows with a last-synced time. (Phone delivery is the eventual target via the trusted-HTTPS tunnel; the mechanism is proven on desktop.)

## Out of scope (explicit)

- Runtime API response caching (deferred D).
- Full offline compose (queuing body edits); only the send request is queued.
- Phone/tunnel deployment + the trusted-HTTPS origin.
- Per-contact / multi-type notifications (one type: "New draft ready"); rich notification actions.
- Notification batching/coalescing.
