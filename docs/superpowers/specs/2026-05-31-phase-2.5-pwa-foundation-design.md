# Phase 2.5 (foundation) — PWA Skeleton + Plumbing — Design

**Date:** 2026-05-31
**Status:** Approved (pending spec review)
**Brief reference:** `BRIEF.md` §2.5 (PWA skeleton + setup wizard), §3 (PWA stack), §12 (PWA design/IA), §9 (the API it consumes), §5 (PWA-to-service auth).

## Context

Phases 0–5 are complete and merged. The service (`apps/service`) exposes the full Phase 3–5 API over HTTPS (`https://localhost:47824`, mkcert cert): auth/session, accounts, threads + `needs-attention`, contacts, drafts (create/edit/send/discard), settings, push, and the SSE stream (`/api/v1/events`). Everything built so far is **API-only** — `apps/service/pwa/` is just a placeholder `index.html` (the service reads it and serves it at `GET /`).

Phase 2.5 was originally slated as "PWA skeleton + setup wizard" right after Phase 2, but was skipped; we're building it now, after Phases 3–5 added the API surface it will surface. Because the full PWA (every §12 screen + setup wizard) is the largest subsystem in the plan, **it is decomposed**: this phase is the **foundation only** — the build/serve/dev pipeline, the app shell (routing + bottom nav), the API client, the auth handshake, SSE→cache plumbing, and the PWA manifest + service worker. The actual designed screens (Needs Attention cards, thread/draft review, etc.) are a **follow-on phase**; route bodies are placeholders here.

### Resolved design decisions

1. **Foundation only** — placeholder route bodies (each calls its API + shows a minimal list/empty state to prove the data path), no designed screens. The setup wizard (§2.5 item 4) is deferred. Web Push (§5.5), runtime API caching, and the offline action queue are deferred — the service worker only precaches the app shell here.
2. **Routing: Wouter** (tiny, hook-based) over React Router — smaller PWA bundle.
3. **Serve the built SPA from Fastify** via `@fastify/static` (a new service dep) + an SPA fallback for non-`/api/v1` GETs. Same origin as the API, so the existing strict CORS is unaffected.
4. **Auth UX:** session token from `localStorage` → else a `#bootstrap=<token>` URL hash (the eventual tray-launch path) → else a "paste your bootstrap token" screen (manual/dev). No Electron-tray change this phase.
5. **`apps/service/pwa` becomes a workspace package** `@secretary/pwa` (added to `pnpm-workspace.yaml`). UI is manually verified (BRIEF §18); the pure plumbing (api client, auth, SSE mapping) is unit-tested with Vitest + jsdom.

## Goals (this phase)

1. A Vite + React 18 + TypeScript + Tailwind app at `apps/service/pwa/` (workspace package `@secretary/pwa`) with `dev`/`build`/`typecheck`/`test` scripts.
2. App shell: Wouter routing + the §12 bottom nav (Needs Attention, Follow-ups, Inbox, Contacts, Settings), mobile-first single-column (centered max-width 720px on desktop, 44pt tap targets), with **placeholder** route bodies that exercise their API.
3. A typed API client (fetch wrapper + TanStack Query hooks) over the `{data}`/`{error}` envelope, reusing `@secretary/shared-types`.
4. The auth handshake (localStorage → hash → paste fallback) → `POST /api/v1/auth/session`.
5. SSE `EventSource` manager (auto-reconnect + backoff) mapping server events → TanStack Query invalidations.
6. PWA `manifest.webmanifest` + Workbox-generated service worker (precache the app shell) + 192/512 icons.
7. The Fastify service serves the built SPA (`@fastify/static` + SPA fallback); the dev loop uses the Vite dev server with an API/SSE proxy.

## Non-goals (deferred)

- All designed screens — Needs Attention cards, thread view, draft review/edit/send UI, Contacts, Follow-ups, Settings forms, voice input, swipe gestures, long-press send (the **next** phase consumes the API client + shell built here).
- The setup wizard (Cloudflare/gateway/VAPID entry) — your machine is already configured; a fresh-install wizard is a later concern.
- **Web Push** subscription + notifications (Phase 5.5); runtime API response caching + offline POST queue + offline banner (5.5).
- Electron-tray changes (the tray "Open Secretary" → browser handshake stays as-is; the paste fallback covers auth meanwhile).
- electron-builder packaging of the PWA (Phase 10).

## Architecture

### Workspace + toolchain

- `pnpm-workspace.yaml` gains `- 'apps/service/pwa'` (it's nested under `apps/service`, so `apps/*` doesn't match it).
- `apps/service/pwa/package.json` (`@secretary/pwa`, private, `type: module`): deps `react`, `react-dom`, `wouter`, `@tanstack/react-query`, `@secretary/shared-types` (`workspace:*`); devDeps `vite`, `@vitejs/plugin-react`, `vite-plugin-pwa` (Workbox), `tailwindcss`/`postcss`/`autoprefixer`, `typescript`, `vitest`, `jsdom`, `@testing-library/react` (for any light component test), `@types/react`/`@types/react-dom`. Scripts: `dev` (`vite`), `build` (`tsc -p tsconfig.json && vite build`), `typecheck` (`tsc -p tsconfig.json --noEmit`), `test` (`vitest run`).
- `apps/service/pwa/tsconfig.json` — a Vite/React config (does NOT extend the Node `tsconfig.base.json`): `target` ES2022, `lib` `["ES2022","DOM","DOM.Iterable"]`, `module` ESNext, `moduleResolution` bundler, `jsx` react-jsx, `strict` + `noUncheckedIndexedAccess` + `exactOptionalPropertyTypes` (match the repo's strictness), `noEmit`, `types: ["vite/client","vite-plugin-pwa/client"]`.
- No native modules → no `better-sqlite3` ABI concern. No `.eslintrc` for the PWA this phase (it has no `lint` script, so `pnpm -r lint` skips it); correctness is via `tsc` typecheck + manual UI verification. (Airbnb-react lint can be added later.)

### Build, serve, and dev loop

- **`vite.config.ts`:** `@vitejs/plugin-react`; `VitePWA` (Workbox `generateSW`, precache the shell, `manifest` inline — name/short_name "Secretary", `display: standalone`, `start_url: /needs-attention`, theme/background colors, 192/512 icons); `build.outDir: 'dist'`; `server`: port 5173, `proxy` for `/api/v1` → `{ target: 'https://localhost:47824', changeOrigin: true, secure: false }` (the SSE route `/api/v1/events` is under that prefix and proxies as a long-lived response).
- **Production serve (service):** add `@fastify/static` to `apps/service`. In `server.ts`, when `pwaDir` is set, register `@fastify/static` rooted at `pwaDir` (serving `index.html` + hashed `/assets/*` + `manifest.webmanifest` + the SW), and add a `setNotFoundHandler`/wildcard that returns `index.html` for any **non-`/api/v1`** GET (SPA client-side routing) while keeping the existing `{error}` envelope for unknown `/api/v1` routes. The existing `onRequest` auth guard already skips non-`/api/v1` paths, so static assets + the SPA fallback are unauthenticated (correct — the bundle is public; the session token gates the API).
- **`index.ts`:** point `pwaDir` at the built SPA — `join(here, '..', 'pwa', 'dist')` (was `join(here, '..', 'pwa')`). In dev you use the Vite dev server instead; in production the service serves `pwa/dist`.
- **Build wiring:** the service `build` script's existing `cpSync('pwa','dist/pwa',...)` is replaced by building the PWA and copying `pwa/dist` → `dist/pwa` (or the service depends on `@secretary/pwa build`). The dev loop (manual verification) uses Vite dev + `dev:server`; production packaging is Phase 10.

### App shell — `src/`

- `src/main.tsx` — React root; wraps `<App/>` in a `QueryClientProvider` (TanStack Query) and registers the service worker (via `virtual:pwa-register` from vite-plugin-pwa). Imports `styles/index.css` (Tailwind).
- `src/App.tsx` — the auth gate + the shell. If not authenticated → render `<AuthGate/>`; else render the router (`wouter`) with the route table + `<BottomNav/>`. Default/redirect route → `/needs-attention`.
- `src/components/AppShell.tsx` + `BottomNav.tsx` — single-column layout (max-width 720px, centered), a top title bar, the content `<main>`, and a fixed bottom nav (5 items, icon + label, 44pt targets, active-route highlight via Wouter's `useRoute`/`useLocation`).
- `src/routes/` — `NeedsAttention.tsx`, `FollowUps.tsx`, `Inbox.tsx`, `Contacts.tsx`, `Settings.tsx`, `ThreadView.tsx` — **placeholder** bodies: each uses its query hook and renders a minimal list / empty / loading / error state (e.g., NeedsAttention lists thread subjects + state; ThreadView shows the thread's messages). Enough to prove the data path; full design is the next phase.

### API client + auth — `src/api/`

- `src/api/client.ts` — `apiFetch<T>(path, opts)`: prefixes `/api/v1`, injects `Authorization: Bearer <session>` from the session store, sets JSON content-type, parses the `{data}`/`{error}` envelope, and throws a typed `ApiError { code, message, status }` on `!ok` or an `{error}` body. A `401` clears the stored session and surfaces an auth error (so the app returns to the gate).
- `src/api/session.ts` — session-token store (localStorage key `secretary.session`), `getSession`/`setSession`/`clearSession`, plus `bootstrapFromHashOrThrow()` (read `#bootstrap=`, `POST /auth/session`, store, clear hash) and `exchangeBootstrap(token)` for the paste path.
- `src/api/hooks.ts` (or per-family files) — TanStack Query hooks: `useNeedsAttention`, `useThread(id)`, `useThreads(filters)`, `useContacts`, `useSettings`, and mutations `useCreateDraft`/`useEditDraft`/`useSendDraft`/`useDiscardDraft`, `useSetState`, `useClassify`, `usePatchContact`. Query keys are namespaced (`['needs-attention']`, `['thread', id]`, …) so the SSE layer can invalidate precisely. (Hooks exist now; the placeholder routes use a subset, the rest are consumed by the next phase's screens.)
- `src/api/AuthGate.tsx` — minimal screen: if a `#bootstrap` hash is present it auto-exchanges; otherwise a single input to paste the bootstrap token (from `~/.secretary/bootstrap-token.txt`) → `exchangeBootstrap`. On success, the app re-renders authenticated.

### SSE — `src/sse/`

- `src/sse/events.ts` — `createEventStream(queryClient)`: opens `EventSource('/api/v1/events')`, on each event maps `type` → `queryClient.invalidateQueries(...)`: `thread:updated` → `['needs-attention']` + `['thread', threadId]` (+ `['threads']`); `draft:ready` → `['thread', threadId]` (+ a `['draft', draftId]` if used); `account:status` → `['accounts']`; `sync:progress` → a status atom. Auto-reconnects with capped exponential backoff on `error`/close; exposes a `connected` status. The pure mapping fn (`eventToInvalidations(event)`) is unit-tested; the `EventSource` wiring is thin.
- A small `useServerEvents()` hook starts the stream once under the authenticated app and tears it down on unmount.

> Note: the SSE endpoint requires the bearer token, but `EventSource` cannot set headers. The service's SSE route auth needs to accept the token another way for the browser — **this phase resolves that**: the SSE manager passes the session token as a query param (`/api/v1/events?token=<session>`), and a tiny service-side change lets the events route accept `?token=` in addition to the `Authorization` header (validated the same way). This is the one additional small server change beyond static serving.

### Manifest + service worker

- Generated by `vite-plugin-pwa` (Workbox `generateSW`): precache the built shell (`index.html` + JS/CSS), `navigateFallback` to `index.html`, `start_url: /needs-attention`, `display: standalone`. 192×192 + 512×512 monogram PNG icons committed under `pwa/public/`. No `runtimeCaching`/push here (5.5).

## Data flow

Dev: browser → Vite dev `:5173` → (proxy) → `https://localhost:47824/api/v1/*` + SSE. Prod: browser → `https://localhost:47824` → Fastify serves `pwa/dist` (shell) + `/api/v1/*`. Auth: load → localStorage session (else hash, else paste) → bearer on every `apiFetch` + `?token=` on the SSE. Reactivity: a server event → `EventSource` → `eventToInvalidations` → TanStack Query refetches the affected keys → placeholder routes (and next phase's screens) update live.

## Error handling

- `apiFetch` throws typed `ApiError`; query hooks expose `error` for the routes to render an error state; `401` clears the session → back to `AuthGate`.
- The SSE manager reconnects with backoff and never throws into React; a disconnected state is surfaced (a banner is a 5.5 concern, but the status exists).
- The auth gate handles a failed/expired bootstrap (single-use) by showing the paste screen with the error.
- The SPA fallback never shadows `/api/v1` (unknown API routes keep their `{error}` 404).

## Testing strategy

Vitest (jsdom environment) in `@secretary/pwa`, covering the **pure plumbing** only (BRIEF §18 — UI manually tested in v1):

- `apiFetch` — unwraps `{data}`, throws `ApiError` on `{error}`/non-2xx, injects the bearer, clears session + throws on 401 (mock `fetch`).
- `session` — get/set/clear (mock `localStorage`); `bootstrapFromHash` reads + clears the hash and exchanges; `exchangeBootstrap` stores on success.
- `eventToInvalidations(event)` — pure mapping: each `ServerEvent` type → the expected query-key set (no real `EventSource`).
- (Optional, light) one render smoke test of `<BottomNav/>` with `@testing-library/react` (5 items, active highlight).
- **Manual verification** (`docs/PHASE-2.5-MANUAL-VERIFICATION.md`): run `dev:server` + `pnpm --filter @secretary/pwa dev`; open `http://localhost:5173`; paste the bootstrap token → authenticated; confirm the bottom nav + routes load (NeedsAttention placeholder lists the classified threads via the real API; ThreadView shows messages); trigger a classify/send and watch a route refresh live via SSE; `pnpm --filter @secretary/pwa build` then load `https://localhost:47824` to confirm the service serves the built SPA + the SW precaches the shell (offline shell load).

## Risks & mitigations

- **New toolchain in a Node-first monorepo** — the PWA's bundler/DOM tsconfig is isolated (doesn't extend the Node base); it's a workspace package so `pnpm install`/`-r typecheck`/`-r test` pick it up; no native deps so no ABI interaction.
- **Serving an SPA from Fastify** — `@fastify/static` + a non-`/api/v1` fallback; the auth guard already excludes non-API paths; verified by loading the built app at `:47824`.
- **SSE auth via `EventSource`** (no custom headers) — resolved with a `?token=` query param + a small events-route change accepting it; the token is a local bearer over localhost HTTPS.
- **Dev CORS** — avoided entirely by the Vite proxy (browser is same-origin to `:5173`); production is same-origin to `:47824`.
- **mkcert cert in the Vite proxy** — `secure:false` on the proxy target (the cert is locally trusted but the proxy is server-side).
- **Scope creep into screens** — explicitly fenced: placeholder route bodies only; the designed screens are the next phase.
